\section{Introduction}
The user interfaces of Decentralized Finance (DeFi) are shifting from manually executing on‐chain transactions to conversational AI agents, ranging from wallet‑embedded copilots, Telegram trading bots, to fully autonomous DeFi assistants. These systems translate natural‑language directives into executable blockchain operations, effectively inserting a probabilistic planning layer into what was a deterministic pipeline under user's control.

The shift introduces an unexamined security boundary. Unlike explicit and formally analyzable smart contracts, large language models (LLMs) exhibit stochasticity and are susceptible to various linguistic and contextual adversarial actions, including prompt injection, indirect influence attacks, contextual contamination, and social‑engineering‑driven specification drift. The threat is amplified in open multi‑party communication environments such as Telegram or WhatsApp, where adversaries may directly interact with the same agent instance as the legitimate user, embedding payloads designed to subvert the transaction‑generation process.

While the existing literature extensively investigates vulnerabilities in smart‑contract correctness, MEV extraction dynamics, oracle manipulation, and private‑key security, there is little systematic analysis of the security properties of novel LLM‑driven transaction‑planning pipelines under adversarial language conditions. The research addresses the gaps by implementing and empirically evaluating a minimal, adversarially tested prototype intersecting AI agents and DeFi infrastructure.

\section{Literature Review}
%

The project integrates DeFi, AI-driven agents, and adversarial machine learning security, encompassing multiple unrelated and evolving fields. To situate the present work, this section connects three major bodies of literature, namely DeFi protocol and transaction-security research, adversarial language-model safety, and agentic LLM pipelines.

\subsection{DeFi Protocols Research}

Mainstream DeFi security research revolves around deterministic smart-contract vulnerabilities, including integer-overflow bugs, faulty permission, and MEV extraction methodologies assuming explicit, analyzable code as the attack surface, treating swaps and liquidity operations as deterministic execution traces \cite{10.1016/j.eswa.2025.128431}. The literature on non‑custodial wallet design and safe transaction construction further emphasizes invariant‑preserving interfaces and formal verification \cite{10.1016/j.eswa.2025.128431, ERINLE2025111691}. All the discussed work share an implicit assumption that the transaction pipeline is audit-able and deterministic.

However, probabilistic LLM-driven planners invalidate such an assumption and introduce stochastic decision chains and potential misalignment between authentic user intents and generated transactions, creating a novel security boundary that warrants further examination.

\subsection{Adversial LLM insecurity}

On top of stochasticity, the weaker adversarial robustness of LLM also induce challenges for implementing a secure swap agent. Recent research has been conducted throughout the document and has demonstrated attacks such as prompt injection, jailbreaks, and indirect prompt attacks \cite{pathade2025redteamingmindmachine}. In particular, research on prompt injection attacks detail how LLMs can be coerced into overriding internal instructions, execution constraints, or system‑level guardrails, especially when they are unable to enforce reliable separation between trusted and insecure channels \cite{pathade2025redteamingmindmachine, li2025securityconcernslargelanguage}. 

Consistent with the research, production-level incident of unprotected LLM agent getting prompt-injected in online forums and sending funds to unauthorized accounts emerged, highlighting realistic vulnerabilities to direct injection, structural weaknesses in LLM trust boundaries, and the ease of hijacking model behavior in open environments\cite{balaha2026}. This aligns with the threat model facing conversational DeFi agents operating in public channels like Telegram, where adversaries co‑exist with legitimate users and strategically inject malicious instructions into shared context.


\subsection{LLM Agents Tools and Deterministic Policy Layer}
LLM agents that use external tools, APIs, and multi-step planners consistently exhibit novel modes of safety failure beyond typical content-moderation issues. Assessments such as Agent-SafetyBench show that no agent achieves a safety score of 60\% in tool invocation, while datasets like ToolSafety further demonstrates that multi-step and indirect harm scenarios significantly reduce the reliability of LLMs \cite{zhang2024agent, xie-etal-2025-toolsafety}. Even safety‑tuned models err in tool‑mediated workflows—misinterpreting intermediate data, hallucinating execution steps, or over‑relying on uncertain tool outputs. These results highlight the need for deterministic policy layers, strict schema validation, and environment constraints rather than relying solely on prompt‑based guardrails.

\section{Application Setting}

\subsection{System Architecture}
\subsubsection{Purpose and Scope}
The system provides a conversation swap planning agent that converts natural-language-based user intents into a deterministic, unsigned transaction plan (\texttt{TxPlan}) for a DeFi swap while the agent never signs or broadcasts transactions. Security objectives are user fund safety, privacy, and policy adherence in adversarial environment (e.g., public chat channels and unsafe data sources). 

\subsubsection{System Components Overview}
The following provides an overview of the components and their interactions.

\begin{figure}[H]
  \centering
  \includegraphics[width=3.3in]{figure/component_diagram.png}
  \caption{System Components Overview (zoom in for details)}
  \label{fig_model}
\end{figure}

A legitimate user interacts with the system via a conversation request to a chat interface (typically a Telegram bot), calling the agent plan API. After the Layer 1 (L1) agent parses and sanitizes the request, it extracts a structured swap intent and retrieves market data. The Layer 2 (L2) policy engine verifies whether the request compiles with deterministic safety and \texttt{allowlist} rules, and returns an unsigned \texttt{TxPlan} marked with \texttt{NEEDS\_OWNER\_SIGNATURE} The user reviews the plan and, if satisfied, signs with their own private key.

In this paper, L1 is the DeFi LLM agent. It is a probabilistic layer responsible for handling input security, filtering requests, and controlling tool invocations, thereby mitigating natural language attacks. Meanwhile, L2 is the policy engine. It is a strict and deterministic enforcement layer that blocks any rule-breaking actions by the LLM or user requests.

\subsubsection{Important Trust and Signer Boundary}
\begin{itemize}
    \item Invariant I1 (Non‑Custodial): No module can produce a signed transaction or trigger an on‑chain side effect.
    \item Invariant I2 (Deterministic Governance): L2 policy decisions dominate LLM/tool outputs.
\end{itemize}

Thus, the agent's possible outputs are limited as follows.
\begin{multicols}{2}
\begin{itemize}
    \item \texttt{REJECTED} (by L1)
    \item \texttt{NEEDS\_OWNER} \\ \texttt{\_SIGNATURE}
    \item \texttt{BLOCKED} (by L2)
    \item \texttt{ERROR} (API error)
\end{itemize}
\end{multicols}

\subsection{Threat Model}
\subsubsection{Assets and Objectives}
The most important assets are user funds and privacy information. The system is supposed to prevent unsafe plans that might incur losses while protecting addresses, balances, and user intentions, and ensuring policy adherence and invariant preservation for system integrity.

\subsubsection{Adversaries and Capabilities}
Assume adversaries can access the input interface in an open communication platform, observe timing/metadata, and influence tool output to the extent allowed by mocked/external data sources. The following adversaries prevail under these assumptions.

\begin{itemize}
    \item External prompt adversary: use chat input to induce unsafe plans (e.g., "I need money", role-play jail breaking). 
    \item Indirect data adversary: inject malicious instructions via external tools/data (e.g., payload in market feeds). 
    \item Context Poisoner: forge provenance (fake admin/system claims) or taint historical context/memory.
    \item Fraudster: exploits policy gaps (e.g., routers/tokens not on allowlists, extreme slippage).
\end{itemize}

\subsubsection{Adversaries Attack Surfaces}

\begin{itemize}
    \item Open chat input channel (Telegram/WhatsApp natural language inputs).
    \item External tool outputs (e.g., quotes and market snapshots are susceptible to indirect injection). 
    \item State/memory (if any retained context is used beyond a single request) allowing information leakage.
    \item Public or unprotected API surface (HTTP endpoints).
\end{itemize}

\subsubsection{Full Security Invariants and Policies} Besides I1 and I2, we introduce two more invariants for schema and context.

\begin{itemize}
    \item Invariant I3 (Schema Integrity): All cross‑component messages conform to \texttt{schemas.py} and are validated before sending and upon arrival.
    \item Invariant I4 (Context Discipline): Tool outputs treated as untrusted data; never merged into instruction channels.
\end{itemize}
