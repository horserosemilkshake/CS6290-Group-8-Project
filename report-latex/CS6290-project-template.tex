\begin{abstract}
Large language model (LLM) agents are beginning to mediate DeFi workflows, but
their security properties remain poorly understood in adversarial language
settings. We present a minimal non-custodial swap agent that converts natural
language requests into unsigned transaction plans and evaluates them through
three layers: L1 LLM guardrails, an L2 deterministic policy engine, and an
optional L3 on-chain validator. We evaluate the system on a fixed 125-case
benchmark comprising four adversarial prompt vectors and a benign baseline.
L1+L2 reduces Attack Success Rate (ASR) from 84\% in the bare configuration to
0\% while maintaining a 0\% false-positive rate. L3 does not further reduce
ASR on this dataset, but it provides immutable defense-in-depth against
application-layer misconfiguration. The results show that a secure DeFi agent
must treat LLM reasoning as untrusted and place deterministic policy
enforcement on the critical path before the owner-signature handoff.
\end{abstract}


\section{Introduction}
DeFi interfaces are shifting from manual wallet interactions to conversational
agents embedded in wallets, bots, and copilots. These systems translate natural
language into transaction plans, inserting a probabilistic planning layer into
what was previously a deterministic user-controlled pipeline.

That shift creates a new security boundary. Smart contracts remain explicit and
auditable, but LLM planners are stochastic and vulnerable to prompt injection,
indirect instruction attacks, context poisoning, and social-engineering-driven
specification drift. The risk is amplified in open channels such as Telegram,
where legitimate users and adversaries can interact with the same agent-facing
surface.

Most prior work studies smart-contract bugs, MEV dynamics, oracle manipulation,
or key management. Much less work examines the security of LLM-mediated
transaction planning under adversarial language conditions. This paper
addresses that gap through a minimal, adversarially evaluated DeFi swap agent
with layered defenses and an explicit non-custodial signer boundary.

\section{Literature Review}

This work sits at the intersection of three literature streams: DeFi
transaction security, adversarial LLM safety, and tool-using LLM agents. We
review each stream to identify the gap addressed by our architecture.

\subsection{DeFi Protocols Research}

Mainstream DeFi security research focuses on deterministic smart-contract
vulnerabilities, such as arithmetic bugs, faulty permissions, and MEV-related
execution risks, under the assumption that the transaction pipeline is explicit
and analyzable \cite{10.1016/j.eswa.2025.128431}. Work on non-custodial wallet
design and safe transaction construction likewise emphasizes
invariant-preserving interfaces and formal verification
\cite{10.1016/j.eswa.2025.128431, ERINLE2025111691}. Across these lines of
work, the transaction planner is usually treated as deterministic.

LLM-driven planners break that assumption. They introduce stochastic decision
chains and possible misalignment between authentic user intent and generated
transactions, creating a new security boundary that classical DeFi analysis
does not cover well.

\subsection{Adversarial LLM Insecurity}

LLMs also have limited adversarial robustness. Prior work documents prompt
injection, jailbreaks, and indirect prompt attacks
\cite{pathade2025redteamingmindmachine}. In particular, prompt-injection
research shows how models can be induced to override internal instructions,
execution constraints, or system-level guardrails when trusted and untrusted
channels are not reliably separated
\cite{pathade2025redteamingmindmachine, li2025securityconcernslargelanguage}.

Reported incidents involving weakly protected LLM agents further suggest that
direct injection is not only a theoretical concern \cite{balaha2026}. These
failures align with the threat model of conversational DeFi agents in public
channels, where adversaries and legitimate users share the same interaction
surface and can compete to influence the planner.


\subsection{LLM Agents Tools and Deterministic Policy Layer}
LLM agents that use external tools, APIs, and multi-step planners show safety
failures beyond conventional content moderation. Benchmarks such as
Agent-SafetyBench and ToolSafety report weak reliability in tool invocation and
multi-step harmful scenarios \cite{zhang2024agent, xie-etal-2025-toolsafety}.
Even safety-tuned models can misread intermediate data, hallucinate execution
steps, or over-trust uncertain tool outputs. These results motivate
deterministic policy layers, strict schema validation, and explicit environment
constraints instead of relying on prompt-only defenses.

\subsection{Synthesis}

\begin{table}[H]
\centering
\caption{Summary of Research Domains, Targets, Limitations}
\label{tab:literature-summary}
\begin{tabular}{lll}
\hline
Domain & Address & Does not address \\ \hline
DeFi security  & \makecell{deterministic\\code\\vulnerabilities}    & \makecell{language misinterpretation\\and adversarial manipulation\\at the prompt layer}      \\ \hline

Our prototype addresses this intersection of blind spots by showing how LLM
uncertainty couples with deterministic financial execution. It proposes
system-level invariants that combine non-custodial guarantees with AI-safety
and adversarial-robustness principles. The resulting threat surface is neither
a classical smart-contract bug nor a standard LLM-safety problem; it is a
hybrid security problem that requires a dedicated application architecture.

\section{Application Setting}

\subsection{System Architecture}
\subsubsection{Purpose and Scope}
The system is a conversational swap-planning agent that converts natural-
language user intents into an unsigned transaction plan (\texttt{TxPlan}) for a
DeFi swap. The agent never signs or broadcasts transactions. Its security goals
are fund safety, privacy preservation, and deterministic policy adherence in
adversarial environments such as public chat channels and untrusted data
sources.

\subsubsection{System Components Overview}
Figure~\ref{fig_components} summarizes the system boundary and the
owner-signature handoff.

\begin{figure}[H]
  \centering
  \includegraphics[width=\linewidth]{figure/architecture_overview_v2.png}
  \caption{System boundary, trusted agent space, and owner-signature handoff.}
  \label{fig_components}
\end{figure}

A legitimate user submits a request through a chat interface, typically a
Telegram bot, which forwards the request to the planning API. L1 sanitizes the
input, extracts a structured swap intent, and coordinates quote retrieval. L2
then checks the plan against deterministic safety rules and allowlists. If the
request passes, the system returns an unsigned \texttt{TxPlan} that requires
owner signature. The user must review and sign the plan with their own wallet.

The figure emphasizes three architectural boundaries. The \emph{trusted agent
space} contains every component that can parse intent or construct a plan. The
\emph{safety gate} marks the last deterministic validation boundary before any
plan reaches the user. The \emph{wallet domain} remains outside the agent: the
system hands off an unsigned plan, and the user alone decides whether to sign.

In our implementation, L1 is the probabilistic language-facing layer. It
handles input filtering, prompt-injection rejection, and output validation. L2
is the deterministic enforcement layer. It blocks requests that violate token,
router, slippage, value-cap, approval, or chain-scope policies, even if the LLM
or an external tool proposes them.

\begin{figure}[H]
  \centering
  \includegraphics[width=\linewidth]{figure/layered_control_flow_v2.png}
  \caption{Layered control flow from natural-language input to owner-signature handoff.}
  \label{fig_layers}
\end{figure}

Figure~\ref{fig_layers} shows how the three control layers divide
responsibility:

\begin{enumerate}
    \item L1 (Input/Output Guardrails): prompt-injection filtering, unsafe instruction rejection, and response validation.
    \item L2 (Deterministic Policy Engine): rule-based checks on token/router allowlists, slippage, transaction value cap, approval patterns, transaction structure, and chain scope.
    \item L3 (Optional On-Chain Validator): pre-flight contract validation (\texttt{Swap\allowbreak Guard}) via \texttt{eth\_call} for selected hard constraints.
\end{enumerate}

\subsubsection{Trust and Signer Boundary}
\begin{itemize}
    \item Invariant I1 (Non-Custodial): No module can produce a signed transaction or trigger an on-chain side effect.
    \item Invariant I2 (Deterministic Governance): L2 policy decisions dominate LLM/tool outputs.
\end{itemize}

The agent is limited to the following observable outcomes.
\begin{multicols}{2}
\begin{itemize}
    \item \texttt{REJECTED} (by L1)
    \item \texttt{NEEDS\_OWNER} \\ \texttt{\_SIGNATURE}
    \item \texttt{BLOCKED} (by L2)
    \item \texttt{ERROR} (API error)
\end{itemize}
\end{multicols}

\subsection{Threat Model}
\subsubsection{Assets and Objectives}
The primary assets are user funds, privacy-sensitive information, and intent
integrity. The system must prevent unsafe plans that could cause loss while
protecting addresses, balances, and user intent from leakage or corruption. It
must also preserve policy adherence and the system invariants that enforce the
non-custodial boundary.

\subsubsection{Adversaries and Capabilities}
We assume adversaries can access the input interface in an open communication
environment, observe timing and metadata, and influence tool output to the
extent allowed by mocked or external data sources. Under that model, the main
attacker classes are:

\begin{itemize}
    \item External prompt adversary: uses chat input to induce unsafe plans, including jailbreaks and emotional or authority-based manipulation \cite{balaha2026}.
    \item Indirect data adversary: injects malicious instructions through external tools or data feeds.
    \item Context poisoner: forges provenance, such as fake admin or system claims, or contaminates retained conversational context.
    \item Fraudster: exploits policy gaps through disallowed routers, tokens, approvals, or extreme execution parameters.
\end{itemize}

\subsubsection{Attack Surfaces}

\begin{itemize}
    \item Open chat input channel (Telegram/WhatsApp natural language inputs).
    \item External tool outputs (e.g., quotes and market snapshots are susceptible to indirect injection). 
    \item State/memory (if any retained context is used beyond a single request) allowing information leakage.
    \item Public or unprotected API surface (HTTP endpoints).
\end{itemize}

\subsubsection{Additional Security Invariants}
Besides I1 and I2, we define two more invariants for schema and context.

\begin{itemize}
    \item Invariant I3 (Schema Integrity): All cross-component messages conform to \texttt{schemas.py} and are validated before sending and upon arrival.
    \item Invariant I4 (Context Discipline): Tool outputs treated as untrusted data; never merged into instruction channels.
\end{itemize}

\section{Implementation}

This section describes the concrete realization of the three defense layers
and the evaluation harness.

\subsection{Technology Stack}

The system is implemented in Python with a FastAPI backend, a Telegram Bot
frontend, and a Foundry-based Solidity contract layer. The agent core resides
in \path{agent_client/src/}, the deterministic policy engine in
\path{policy_engine/}, and the on-chain validator in
\path{contracts/src/SwapGuard.sol}. Benchmarking is orchestrated by a smoke
harness (\path{harness/runner.py}) that replays fixed test suites against
configurable defense profiles.

\subsection{L1 Guardrails}

L1 is split into an \texttt{Input\allowbreak Guardrail} and an
\texttt{Output\allowbreak Guardrail} (\path{agent_client/src/agents/l1_agent.py}).
The input guard maintains fourteen regex-based blocked patterns (e.g.,
\textit{``ignore all instructions,''} \textit{``system prompt,''}
\textit{``bypass guardrail''}) and three encoded-content patterns (base64,
rot13, unicode escapes). It also enforces a 500-character limit, swap-keyword
presence, and PII scrubbing (Ethereum addresses, private-key mentions). The
output guard validates JSON schema conformance of the LLM planner, checks for
forbidden tool calls (\texttt{broadcast\_transaction},
\texttt{sign\_transaction}, \texttt{approve\_unlimited}), and blocks privacy
leaks such as transaction hashes.

\subsection{L2 Deterministic Policy Engine}

L2 is implemented as a pure-Python rule engine with zero LLM dependencies
(\path{policy_engine/engine.py}). It evaluates eleven deterministic rules
against every \texttt{Swap\allowbreak Intent} and \texttt{Quote\allowbreak Response}:

\begin{itemize}
    \item R-01 Token allowlist
    \item R-02 Router allowlist
    \item R-03 Slippage limit
    \item R-04 Per-transaction value cap
    \item R-05 No unlimited approvals
    \item R-07 Transaction-plan structure validation
    \item R-09 Quote expiry enforcement
    \item R-13 Unsafe request overrides
    \item R-16 Trusted market snapshot prerequisite
    \item R-17 Network scope enforcement
    \item R-23 Request-shape sanity checks
\end{itemize}

A central architectural invariant is that the policy decision is
\textit{non-overridable}: if any rule fires, the engine returns \texttt{BLOCK}
regardless of what the LLM or any tool suggests.

\subsection{L3 On-Chain Validator}

L3 consists of the \texttt{Swap\allowbreak Guard} Solidity contract
(\path{contracts/src/SwapGuard.sol}) and a zero-dependency Python validator
(\path{policy_engine/l3_validator.py}). The contract mirrors four hard
constraints (R-01, R-02, R-03, R-04) and exposes a \texttt{validateSwap()}
method. The Python side constructs the \texttt{eth\_call} payload using a
pure-stdlib Keccak-256 implementation, avoiding any external Web3 library
dependency. This design lets L3 run in resource-constrained or air-gapped
audit environments.

\section{Experiment and Methodology}

This section describes the benchmark used to evaluate the adversarial
robustness of the proposed swap agent. The design goal is to isolate the
contribution of each defense layer while preserving reproducibility.

\subsection{Benchmark Design}

We evaluate prompt-driven and tool-mediated attacks that target agent behavior
before any on-chain submission. The fixed v2 benchmark
(\path{final_attack_dataset_v2}) comprises 125 cases: 100 adversarial
cases evenly distributed across four attack vectors and 25 benign swap
requests to measure false positives. The four attack vectors are:

\begin{itemize}
    \item \textbf{Direct injection:} explicit jailbreak, role-play, or system-instruction override attempts.
    \item \textbf{Indirect / encoded:} obfuscated or encoded malicious intent (e.g., base64, unicode escapes, semantic reframing).
    \item \textbf{Tool poisoning:} manipulation of tool-related assumptions, such as injecting malicious router addresses, extreme slippage, or unauthorized token symbols inside otherwise valid swap language.
    \item \textbf{Memory poisoning:} context poisoning and fake prior-authority claims that exploit retained conversational state.
\end{itemize}

Figure~\ref{fig_sample_cases} illustrates two representative cases from the
benchmark.

\begin{figure}[H]
  \centering
  \includegraphics[width=3.3in]{figure/Sample test cases.png}
  \caption{Representative benchmark cases illustrating direct-injection and
  tool-poisoning patterns in the v2 dataset.}
  \label{fig_sample_cases}
\end{figure}

\subsection{Metrics and Ablation Setup}

A successful attack is one that induces an unsafe \texttt{ALLOW} decision on an
adversarial case. We report three metrics:

Attack Success Rate (ASR):
\begin{equation}
    \text{ASR}=\frac{\text{adversarial cases with observed}=\texttt{ALLOW}}{\text{total adversarial cases}}
\end{equation}

False Positive Rate (FP):
\begin{equation}
    \text{FP}=\frac{\text{benign cases with observed}\neq\texttt{ALLOW}}{\text{total benign cases}}
\end{equation}

Time Robustness (TR):
\begin{equation}
    \text{TR}=\max_{i}{\text{duration}_i}
\end{equation}

We run the benchmark against four configurations: \texttt{bare}, L1, L1+L2, and
L1+L2+L3. To describe the marginal value of each added defense layer, let
$L(i)$ denote the system with defenses up to layer $i$. The marginal utility
of layer $L(i+1)$ is:
\begin{equation}
    u(i+1) = t(L(i)) - t(L(i+1))
\end{equation}
where $t:(\mathbb{D},\mathbb{A}^4)\to\mathbb{R}^3$ maps a configuration and
attack set to the observed metrics. We use this formulation only to motivate
comparative evaluation.

All experiments use a fixed random seed (6290), a recorded git commit hash,
and a SHA-256 checksum of the test suite to ensure reproducibility.

\subsection{Overall Results}

The final evaluation dataset (\path{final_attack_dataset_v2}) contains 125
cases: 100 adversarial cases evenly distributed across four attack vectors and
25 benign cases.

The main empirical result is simple: L1+L2 reduces ASR from 84\% to 0\%
without increasing false positives. The bare configuration allowed 84\% of
adversarial cases, confirming that an undefended LLM is highly vulnerable in
this setting. L1 reduced ASR to 25\%, and L2 eliminated all remaining
successful attacks on this benchmark. L3 did not further reduce ASR because L2
already blocked every successful adversarial case. The 0\% FP rate shows that
the security gains do not trade off against usability: all 25 benign requests
still reach the owner-signature handoff.

L1+L2 blocks all 100 adversarial cases. The breakdown also clarifies the layer
split: L1 mostly handles direct injection, indirect or encoded attacks, and
memory poisoning through \texttt{REFUSE} outcomes, whereas L2 mainly handles
tool poisoning through \texttt{BLOCK} outcomes. Tool-poisoning requests often
use syntactically valid swap language, so they survive language filtering but
fail deterministic checks on tokens, routers, and execution bounds.

\subsection{Layer-by-Layer Analysis}

\subsubsection{L1 Contribution}
L1 provides the largest single reduction in ASR ($84\% \to 25\%$). It rejects
many prompt injections before planning and validates model outputs before any
plan leaves the LLM-facing layer. On this benchmark, L1 is especially effective
against direct injection, indirect or encoded attacks, and memory poisoning.
However, L1 remains a coarse, probabilistic filter: it cannot catch
parameter-level attacks that use legitimate swap language to smuggle malicious
execution parameters.

\subsubsection{L2 Contribution}
L2 provides the decisive final reduction ($25\% \to 0\%$). It catches plans
that look syntactically valid but violate deterministic rules such as token
allowlists, router allowlists, slippage limits, value caps, approval patterns,
and structural constraints. L2 is particularly important for tool poisoning:
under L1 alone, 20 tool-poisoning cases still pass; under L1+L2, all 20 are
blocked. A key architectural property is that L2 is \textit{non-overridable} by
LLM output. The policy engine evaluates every plan independently, so even a
complete jailbreak of the LLM cannot bypass the hard parameter constraints.

\subsubsection{L3 Contribution}
L3, the on-chain validator (\texttt{Swap\allowbreak Guard}), mirrors key L2
checks in an immutable contract and runs them through a pre-flight
\texttt{eth\_call}. Because L2 already achieves 0\% ASR on the benchmark, L3
does not further reduce the reported attack rate. Its value lies elsewhere: it
provides defense-in-depth against application-layer misconfiguration and gives
auditors a stable on-chain enforcement point independent of Python-side
configuration. The codebase includes a commented-out WBTC misconfiguration
experiment in the L3 validator that illustrates how L3 can catch allowlist
drift that might escape the Python layer.

\subsection{Latency Considerations}

We report maximum request latency $\text{TR}$ for each configuration. The bare
configuration reaches 4.29\,s. L1 lowers the maximum observed latency to
3.14\,s, and L1+L2 reaches 3.08\,s. Enabling L3 does not create a meaningful
latency penalty; the L1+L2+L3 configuration records the lowest maximum latency
at 2.96\,s. These numbers are not statistical noise: early rejection via L1
\texttt{REFUSE} or L2 \texttt{BLOCK} short-circuits expensive LLM and tool
calls, so added defenses can improve both security and latency.

\subsection{Discussion}

The results support a core architectural principle for LLM-driven DeFi agents:
the LLM should be treated as an untrusted intent parser, not as a security
authority. The key boundary is a \textit{probabilistic-to-deterministic
handoff}. L1 performs stochastic language filtering; L2 performs deterministic
policy enforcement on the critical path before the owner-signature handoff.

Bare LLM planning is unsafe, and prompt-only filtering is insufficient. The
benchmark clarifies the role of each layer: L1 handles attacks visible in
language, L2 handles attacks that survive language parsing but fail policy, and
L3 provides operational assurance and integrity mirroring. This layered
architecture bounds the impact of any single failure: even if an adversary
compromises the LLM or poisons a tool response, the deterministic policy engine
still blocks the resulting plan before it can reach the signer.

\section{Limitations}

Several limitations bound the scope of the present evaluation.

\textbf{Benchmark scope.} The evaluation covers only single-turn attacks.
Multi-turn conversational context poisoning, adaptive adversaries that refine
attacks based on prior rejections, and persistent state manipulation are not
represented in the current dataset.

\textbf{Simulated environment.} The smoke harness uses mocked tool responses
rather than live DEX APIs. Real-world factors such as MEV dynamics, network
congestion, quote expiry variance, and aggregator-specific edge cases are not
modeled.

\textbf{Static allowlists.} The token and router allowlists are small and fixed.
A production deployment would require continuous governance of these lists,
which introduces an additional operational attack surface.

\textbf{L3 deployment assumptions.} L3 was evaluated against a local Anvil node.
Mainnet RPC latency, gas costs for contract updates, and the operational
complexity of deploying and maintaining the \texttt{Swap\allowbreak Guard} contract are not
quantified here.

\textbf{Binary success metric.} ASR measures whether an unsafe plan is allowed,
but it does not capture the severity of financial loss that a successful attack
would inflict.

\section{Future Work}

Future work should address the limitations above in five directions:

\begin{enumerate}
    \item \textbf{Multi-turn adversarial dialogue.} Extend the benchmark to stateful, multi-turn interactions where adversaries gradually shift context or embed instructions across benign-looking turns.
    \item \textbf{Real-tool integration benchmarks.} Evaluate the agent against live DEX aggregators (e.g., 1inch, 0x API) to measure robustness against indirect tool-output poisoning and quote-manipulation attacks.
    \item \textbf{Formal verification of policy invariants.} Model the L2 rule set in a formal specification language such as TLA+ to prove that the allowed state space preserves fund safety under all reachable inputs.
    \item \textbf{Adaptive L1 guardrails.} Move beyond static regex patterns to lightweight fine-tuned classifiers or retrieval-augmented detection that adapts to novel prompt-injection templates without re-deploying the entire agent.
    \item \textbf{Governance and allowlist dynamics.} Study the security economics of allowlist updates, including quorum mechanisms, time-locks, and oracle-based token-verification pipelines.
\end{enumerate}

\section{Conclusion}

This paper presented the design, implementation, and adversarial evaluation of a
minimal DeFi swap agent with layered security guardrails. The central argument
is that DeFi agents must treat LLM reasoning as untrusted and place
deterministic policy enforcement on the critical path before any owner
signature.

Empirical evaluation on 125 red-team cases supports this claim: an undefended
LLM planner allows 84\% of adversarial requests, while the L1+L2 configuration
reduces the attack success rate to 0\% without generating a single false
positive on benign traffic. The optional L3 on-chain validator adds
immutability and operational assurance, though its independent value is best
demonstrated through misconfiguration experiments rather than the current
attack benchmark.

The non-custodial invariant remains the ultimate safety bound. Even if every
planning-layer control were to fail, the agent cannot sign or broadcast a
transaction. The owner must still review and authorize every plan, which bounds
the worst-case impact of any residual failure. As conversational interfaces
increasingly mediate financial workflows, this layered, non-custodial
architecture offers a practical path toward adversarially robust AI-driven DeFi.
