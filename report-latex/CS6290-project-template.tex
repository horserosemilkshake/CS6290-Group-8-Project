\begin{abstract}
Large language model (LLM) agents are starting to mediate DeFi workflows, but
their security properties remain unclear when they operate over open,
adversarial natural-language channels. We present a minimal non-custodial DeFi
swap agent that converts user requests into unsigned transaction plans and
evaluates them through three layers: L1 language-facing guardrails, an L2
deterministic policy engine, and an optional L3 on-chain validator. The system
is developed in a specification-driven manner and evaluated on a fixed final
125-case dataset containing 100 adversarial prompts across four attack vectors
plus 25 benign requests. L1+L2 reduces Attack Success Rate (ASR) from 84\% in
the bare configuration to 0\% while maintaining a 0\% false-positive rate. L3
does not further reduce ASR on this benchmark, but a targeted misconfiguration
experiment shows that it can catch configuration drift that escapes the Python
layer. Beyond the benchmark, the prototype externalizes and hash-checks its
system prompt, records tool provenance and quote-validity metadata in each
plan, and exposes explicit owner-signature handoff interfaces that make the
signer boundary observable in both testing and live demos. The main lesson is
that DeFi agents should treat LLM output as untrusted and place deterministic
enforcement on the critical path before the owner-signature handoff.
\end{abstract}


\section{Introduction}
DeFi user interfaces are moving from manual wallet interactions toward
conversational agents embedded in bots, wallets, and copilots. This shift is
convenient for users, but it also inserts a probabilistic planning layer into
what used to be a more explicit and user-controlled transaction pipeline.

That change creates a new security boundary. Smart contracts remain explicit
and auditable, but an LLM planner can be influenced by prompt injection,
context poisoning, encoded instructions, and tool-output manipulation. The risk
is especially important in open communication channels such as Telegram, where
benign users and adversaries can reach the same agent-facing interface
surface~\cite{nizzoli2020charting}.

Most prior work on DeFi security focuses on smart-contract bugs, MEV
dynamics~\cite{li2023demystifying}, oracle manipulation, or wallet key
management. Much less work studies the security of LLM-mediated transaction
planning under adversarial language conditions. Our project addresses that gap
through a minimal DeFi swap agent whose scope is intentionally narrow: it
plans plain swap requests, never signs or broadcasts transactions, and hands
off only unsigned plans to the owner.

This paper makes five concrete contributions. First, we design and implement a
three-layer defense architecture for non-custodial DeFi swap planning. Second,
we show how a specification-driven workflow links requirements, rules, tests,
and final metrics. Third, we expose the signer boundary as an explicit API- and
data-model concept rather than leaving it implicit inside the UI. Fourth, we
evaluate the system on a fixed final 125-case benchmark and complement the main
benchmark with a targeted L3 misconfiguration experiment that clarifies the
value of on-chain defense-in-depth. Fifth, we separate canonical reproducible
benchmarking from live external-tool validation, allowing the same prototype to
support both stable measurement and credible demos.

\section{Literature Review}

This work sits at the intersection of three literature streams: DeFi
transaction security, adversarial LLM safety, and tool-using LLM agents. We
review each stream to identify the gap addressed by our architecture.

\subsection{DeFi Protocols Research}

Mainstream DeFi security research focuses on deterministic smart-contract
vulnerabilities, such as arithmetic errors, faulty permissions, and
MEV-related execution risks, under the assumption that the transaction
pipeline is explicit and analyzable \cite{10.1016/j.eswa.2025.128431}. Work on
non-custodial wallet design and safe transaction construction similarly
emphasizes invariant-preserving interfaces and strong separation between
planning, signing, and execution \cite{10.1016/j.eswa.2025.128431,
ERINLE2025111691}. Across these lines of work, the planner is usually treated
as deterministic.

LLM-based planners break that assumption. They introduce stochastic decision
chains and possible misalignment between authentic user intent and generated
transactions, creating a security boundary that classical DeFi analysis does
not address well.

\subsection{Adversarial LLM Insecurity}

LLMs have limited adversarial robustness. Prior work documents prompt
injection, jailbreaks, and indirect prompt attacks
\cite{pathade2025redteamingmindmachine, derner2024security}. In particular,
prompt-injection research shows that models can be induced to override
instructions, execution constraints, or system-level guardrails when trusted
and untrusted channels are not reliably separated
\cite{pathade2025redteamingmindmachine, li2025securityconcernslargelanguage}.

Reported incidents involving weakly protected LLM agents further suggest that
direct injection is not only a theoretical concern \cite{balaha2026}. These
failures align closely with the threat model of conversational DeFi agents in
public channels, where adversaries and legitimate users share the same
interaction surface and compete to influence the planner.


\subsection{LLM Agents Tools and Deterministic Policy Layer}
LLM agents that use external tools, APIs, and multi-step planners show safety
failures beyond conventional content moderation. Benchmarks such as
Agent-SafetyBench and ToolSafety report weak reliability in tool invocation and
multi-step harmful scenarios \cite{zhang2024agent, xie-etal-2025-toolsafety}.
Even safety-tuned models can misread intermediate data, hallucinate execution
steps, or over-trust uncertain tool outputs. These results motivate
deterministic policy layers, strict schema validation, and explicit
environment-level constraints instead of prompt-only defenses.

\section{Specification-Driven Development}

The project was developed in a specification-driven manner rather than as a
purely prompt-driven prototype. We first defined 18 project-level
specifications (\texttt{S-01} to \texttt{S-18}) covering functionality,
security, benchmarking, and performance. We then mapped these specifications to
a named implementation-rule set (\texttt{R-xx}) and linked those rules to
tests, harness logic, and report metrics. A 35-task development checklist
(\path{docs/specification/task-list-llm.md}) guided the implementation and

an LLM-based financial system because it reduces specification drift: the
report, the code, and the evaluation all refer back to the same named
requirements.

\section{System and Threat Model}

\subsection{System Architecture}
\subsubsection{Purpose and Scope}
The system is a conversational swap-planning agent that converts natural-
language user intents into an unsigned transaction plan (\texttt{TxPlan}) for a
DeFi swap. The agent never signs or broadcasts transactions. Its security goals
are fund safety, privacy preservation, and deterministic policy adherence in
adversarial environments such as public chat channels and untrusted data
sources.

\subsubsection{System Components Overview}
Figure~\ref{fig_components} summarizes the system boundary and the
owner-signature handoff.

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=\linewidth]{figure/architecture_overview_v2.png}
  \caption{System boundary, trusted agent space, and owner-signature handoff.}
  \label{fig_components}
\end{figure}

A legitimate user submits a request through a chat interface, typically a
Telegram bot, which forwards the request to the planning API. L1 sanitizes the
input, parses a structured swap intent, and coordinates quote retrieval. L2
checks the resulting plan against deterministic safety rules and allowlists. If
the request passes, the system returns an unsigned \texttt{TxPlan} that
includes quote validity, slippage bounds, and handoff metadata. The user must
review the plan and decide whether to authorize it in their own wallet domain.

The figure emphasizes three architectural boundaries. The \emph{trusted agent
space} contains every component that can parse intent or construct a plan. The
\emph{safety gate} marks the last deterministic validation boundary before any
plan reaches the user. The \emph{wallet domain} remains outside the agent: the
system hands off an unsigned plan, and the user alone decides whether to sign.

In our implementation, L1 is the probabilistic language-facing layer. It
handles input filtering, prompt-injection rejection, and output validation. L2
is the deterministic enforcement layer. It blocks requests that violate token,
router, slippage, value-cap, approval, or chain-scope policies, even if the
LLM or an external tool proposes them.

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=\linewidth]{figure/layered_control_flow_v2.png}
  \caption{Layered control flow from natural-language input to owner-signature handoff.}
  \label{fig_layers}
\end{figure}

Figure~\ref{fig_layers} shows how the three control layers divide
responsibility:

\begin{enumerate}
    \item L1 (Input/Output Guardrails): prompt-injection filtering, unsafe instruction rejection, schema validation, and basic privacy checks.
    \item L2 (Deterministic Policy Engine): rule-based checks on token/router allowlists, slippage, transaction value cap, approval patterns, transaction structure, quote freshness, request shape, and chain scope.
    \item L3 (Optional On-Chain Validator): pre-flight contract validation (\texttt{Swap\allowbreak Guard}) via \texttt{eth\_call} for selected hard constraints.
\end{enumerate}

\subsubsection{Trust and Signer Boundary}
\begin{itemize}
    \item Invariant I1 (Non-Custodial): No module can produce a signed transaction or trigger an on-chain side effect.
    \item Invariant I2 (Deterministic Governance): L2 policy decisions dominate LLM/tool outputs.
\end{itemize}

At the user-facing level, these implementation details collapse into four
observable outcome classes used throughout the benchmark:
\begin{itemize}
    \item \texttt{REFUSE}: L1 rejects the request before a valid plan exists.
    \item \texttt{BLOCK}: L2 or L3 rejects a plan that violates deterministic policy.
    \item \texttt{ALLOW}: the system returns \texttt{NEEDS\_OWNER\_SIGNATURE} with an unsigned \texttt{TxPlan}.
    \item \texttt{ERROR}: tool, parsing, or runtime failure prevents a valid decision.
\end{itemize}

\subsection{Threat Model}
\subsubsection{Assets and Objectives}
The primary assets are user funds, privacy-sensitive information, and intent
integrity. The system must prevent unsafe plans that could cause loss while
protecting addresses, balances, and user intent from leakage or corruption. It
must also preserve policy adherence and the invariants that enforce the
non-custodial signer boundary.

\subsubsection{Adversaries and Capabilities}
We assume adversaries can access the input interface in an open communication
environment, observe timing and metadata, and influence tool output to the
extent allowed by mocked or external data sources. Under that model, the main
attacker classes are:

\begin{itemize}
    \item External prompt adversary: uses chat input to induce unsafe plans, including jailbreaks and emotional or authority-based manipulation \cite{balaha2026}.
    \item Indirect data adversary: injects malicious instructions through external tools or data feeds.
    \item Context poisoner: forges provenance, such as fake admin or system claims, or contaminates retained conversational context.
    \item Fraudster: exploits policy gaps through disallowed routers, tokens, approvals, or extreme execution parameters.
\end{itemize}

\subsubsection{Attack Surfaces}
\begin{itemize}
    \item Open chat input channel (Telegram-style natural-language inputs).
    \item External tool outputs (e.g., market snapshots and quotes), which must be treated as untrusted data.
    \item State or memory, if any retained context is used beyond a single request.
    \item Public HTTP routes, including runtime control-plane endpoints used
    for testing, demos, and signer-boundary inspection.
\end{itemize}

\subsubsection{Additional Security Invariants}
Besides I1 and I2, we define two more invariants for schema and context.

\begin{itemize}
    \item Invariant I3 (Schema Integrity): All cross-component messages conform to \texttt{schemas.py} and are validated before sending and upon arrival.
    \item Invariant I4 (Context Discipline): Tool outputs treated as untrusted data; never merged into instruction channels.
\end{itemize}

\section{Implementation}

This section describes the concrete realization of the three defense layers,
the signer-boundary implementation, and the evaluation pipeline.

\subsection{Technology Stack}

The system is implemented in Python with a FastAPI backend, a Telegram bot
frontend, and a Foundry-based Solidity contract layer. The agent core resides
in \path{agent_client/src/}, the deterministic policy engine in
\path{policy_engine/}, and the on-chain validator in
\path{contracts/src/SwapGuard.sol}. Benchmarking and artifact generation are
driven by \path{scripts/run_integration_test.py}, which writes final results,
plots, and a report-ready threat-model summary. The repository also includes
real-tool smoke, live benchmark, system-prompt integrity, and L2/L3 policy
parity scripts. Together, these utilities make the prototype more inspectable:
they distinguish canonical evaluation from live integration checks and turn
important security assumptions into executable checks.

\subsection{L1 Guardrails and Prompt Integrity}

L1 is split into an \texttt{Input\allowbreak Guardrail} and an
\texttt{Output\allowbreak Guardrail} (\path{agent_client/src/agents/l1_agent.py}).

The input guard applies five categories of checks before a request reaches
the LLM planner:

\begin{enumerate}
    \item \textbf{Injection pattern matching.} 21 regular-expression patterns organized into five groups: instruction overrides (e.g., ``ignore all instructions'', ``you are now''), policy/guardrail bypass phrases (e.g., ``override policy'', ``bypass guardrail''), code-injection markers (e.g., \texttt{<script>}, \texttt{DROP TABLE}), admin-impersonation claims (e.g., ``verified as admin'', ``on behalf of''), and financial-fraud directives (e.g., ``steal the funds'', ``send fees to my wallet'').
    \item \textbf{Encoding detection.} Three patterns that flag Base64, Rot13, hex-escape (\texttt{\textbackslash x..}), and HTML entity (\texttt{\&\#...;}) content.
    \item \textbf{Length limit.} Messages exceeding 500 characters are rejected.
    \item \textbf{Swap-keyword requirement.} At least one of \texttt{swap}, \texttt{exchange}, \texttt{trade}, \texttt{convert}, \texttt{buy}, or \texttt{sell} must appear.
    \item \textbf{Sensitive-info detection.} Ethereum addresses (40-character hex strings) and private-key or seed-phrase mentions are flagged.
\end{enumerate}

The output guard validates the LLM-produced structure, enforces positive
integer amount format, rejects four forbidden tool references
(\texttt{broadcast\_transaction}, \texttt{sign\_transaction},
\texttt{transfer\_funds}, and \texttt{approve\_unlimited}), and blocks
privacy leaks such as transaction hashes.

The LLM system prompt is also externalized into
\path{agent_client/src/llm/system_prompt.txt} and paired with a recorded hash,
a verification script, and integrity tests. This does not make prompt
tampering impossible, but it does move prompt integrity from an undocumented
inline code string to an auditable repository artifact. In a student project,
that shift matters because it makes one important part of the LLM trust
boundary visible and reviewable.

\subsection{L2 Deterministic Policy Engine}

L2 is implemented as a pure-Python rule engine with zero LLM dependencies
(\path{policy_engine/engine.py}). It evaluates eleven deterministic checks
against every \texttt{Swap\allowbreak Intent} and \texttt{Quote\allowbreak Response}:

\begin{itemize}
    \item R-01 Token allowlist
    \item R-02 Router allowlist
    \item R-03 Slippage limit
    \item R-04 Per-transaction value cap
    \item R-05 No unlimited approvals
    \item R-07 Transaction-plan structure validation
    \item R-09 Quote expiry enforcement
    \item R-13 Unsafe request overrides
    \item R-16 Trusted market snapshot prerequisite
    \item R-17 Network scope enforcement
    \item R-23 Request-shape sanity checks
\end{itemize}

The concrete configuration values used in the final evaluation are:

\begin{itemize}
    \item \textbf{Token allowlist (R-01):} ETH, WETH, USDC, USDT, DAI.
    \item \textbf{Router allowlist (R-02):} 1inch v5, 1inch v6, and the 0x Exchange Proxy.
    \item \textbf{Slippage cap (R-03):} 1000 basis points (10\%).
    \item \textbf{Value cap (R-04):} 5.0 ETH-equivalent per transaction.
    \item \textbf{Chain scope (R-17):} Ethereum mainnet (chain ID 1) and Sepolia testnet (chain ID 11155111).
\end{itemize}

Several of these checks are especially important for the MVP scope. The system
intentionally supports only plain swap intents. User-supplied execution
overrides such as manual slippage, router address, recipient, gas settings, or
price claims are blocked until they have their own verified normalization path.
To support these checks, the implementation extracts request-side signals from
raw user text before evaluation (\path{policy_engine/rules.py},
\texttt{extract\_request\_signals()}) and passes them into the policy engine as
structured metadata. This lets L2 reason not only about the final quote, but
also about suspicious request-shape features such as manual router overrides,
unsupported wei-denominated requests, explicit deadline manipulation, or
untrusted price claims. The resulting design treats language as evidence, not
as authority.

The key architectural invariant is that the L2 decision is
\textit{non-overridable}: if any rule fires, the engine returns
\texttt{BLOCK} regardless of what the LLM or any tool suggests.

\subsection{L3 On-Chain Validator}

L3 consists of the \texttt{Swap\allowbreak Guard} Solidity contract
(\path{contracts/src/SwapGuard.sol}) and a zero-dependency Python validator
(\path{policy_engine/l3_validator.py}). The contract mirrors four hard
constraints (R-01, R-02, R-03, R-04) and exposes a \texttt{validateSwap()}
method. The Python side constructs the \texttt{eth\_call} payload using a
pure-stdlib Keccak-256 implementation, avoiding any external Web3 library
dependency. This design choice keeps the dependency footprint minimal and lets
L3 run in resource-constrained or air-gapped audit environments where
installing a full Web3 stack is impractical or undesirable.

To support that operational role, the repository also includes a policy parity
checker (\path{scripts/check_policy_parity.py}) that compares the deployed
contract's scalar thresholds and allowlist decisions against the current Python
configuration. This script does not replace governance, but it does turn one
important operational assumption---that L2 and L3 still match---into an
explicit check rather than an implicit hope.

\subsection{Signer-Boundary and Provenance Metadata}

When a request passes L1, L2, and optional L3, the system returns a
\texttt{TxPlan} rather than a signed transaction. This object carries not only
the unsigned transaction fields, but also \texttt{slippage\_bounds},
\texttt{quote\_validity}, \texttt{wallet\_handoff}, and \texttt{tool\_audit}
metadata. In the current prototype, the signer boundary is represented by an
explicit in-memory wallet handoff service
(\path{agent_client/src/wallet/bridge.py}) with a configurable TTL (default
300\,s). Table~\ref{tab:api-endpoints} lists the six API endpoints that expose

the published summary numbers still match the underlying case-level reports.

\section{Experimental Methodology}

This section describes the benchmark used to evaluate the adversarial
robustness of the proposed swap agent. The design goal is to isolate the
contribution of each defense layer while preserving reproducibility.

\subsection{Benchmark Design}

We evaluate prompt-driven and tool-mediated attacks that target agent behavior
before any on-chain submission. The final benchmark, stored at
\path{testcases/final_attack_dataset.json}, contains 125 cases: 100
adversarial cases evenly distributed across four attack vectors and 25 benign
swap requests to measure false positives. The four attack vectors are:

\begin{itemize}
    \item \textbf{Direct injection:} explicit jailbreak, role-play, or system-instruction override attempts.
    \item \textbf{Indirect / encoded:} obfuscated or encoded malicious intent (e.g., base64, unicode escapes, semantic reframing).
    \item \textbf{Tool poisoning:} manipulation of tool-related assumptions, such as injecting malicious router addresses, extreme slippage, or unauthorized token symbols inside otherwise valid swap language.
    \item \textbf{Memory poisoning:} context poisoning and fake prior-authority claims that exploit retained conversational state.
\end{itemize}

Figure~\ref{fig_sample_cases} illustrates two representative cases from the
benchmark.

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=3.3in]{figure/Sample test cases.png}
  \caption{Representative benchmark cases illustrating direct-injection and
  tool-poisoning patterns in the final 125-case dataset.}
  \label{fig_sample_cases}
\end{figure}

The benchmark is executed under four configurations: \texttt{bare},
\texttt{l1}, \texttt{l1l2}, and \texttt{l1l2l3}. The canonical numbers in this
report come from the deterministic mock path (\texttt{REAL\_TOOLS=false}),
which isolates the safety architecture from external API instability and makes
the results reproducible. The repository also supports a real-tool path using
CoinGecko and 1inch with strict fail-closed behavior; that path is used for
integration smoke tests, guarded live benchmarking, and demos, not for the
main reported benchmark.

\subsection{Canonical Benchmark vs. Live Integration Validation}

The project deliberately separates two kinds of evidence. The first is the
\emph{canonical benchmark}, which uses the frozen 125-case dataset and
deterministic mock tool responses to measure the defense architecture itself.
The second is \emph{live integration validation}, which turns on
\texttt{REAL\_TOOLS=true} and requires strict fail-closed behavior against real
CoinGecko and 1inch calls. The live path is valuable because it exercises API
configuration, provenance logging, quote-expiry fields, and owner-handoff
metadata under realistic conditions. However, it is not used as the primary
experimental result because its latency and behavior depend on unstable
external services.

\subsection{Metrics and Ablation Setup}

A successful attack is one that induces an unsafe \texttt{ALLOW} decision on an
adversarial case. We report three metrics, following standard adversarial-evaluation conventions~\cite{shen2024anything}:

Attack Success Rate (ASR):
\begin{equation}
    \text{ASR}=\frac{\text{adversarial cases with observed}=\texttt{ALLOW}}{\text{total adversarial cases}}
\end{equation}

False Positive Rate (FP):
\begin{equation}
    \text{FP}=\frac{\text{benign cases with observed}\neq\texttt{ALLOW}}{\text{total benign cases}}
\end{equation}

Time Robustness (TR):
\begin{equation}
    \text{TR}=\max_{i}{\text{duration}_i}
\end{equation}

We use the maximum rather than the mean because the worst-case latency
determines whether any single request can stall the agent under adversarial
conditions. A low TR indicates that even the slowest case completes quickly,
which is important for denial-of-service resistance in an interactive chat
setting.

Here, \texttt{ALLOW} corresponds to the API outcome
\texttt{NEEDS\_OWNER\_SIGNATURE}; the agent has produced an unsigned plan and
paused at the signer boundary. We compare the four configurations by ablation
to isolate the marginal contribution of each layer. The pipeline freezes the
final dataset into version-controlled JSON and regenerates report artifacts
under \path{artifacts/final_results/}, \path{report-latex/figures/}, and
\path{docs/threat-model/final_threat_model.md}. The same pipeline also writes a
validation report that checks the consistency of summary metrics against the
case-level reports before those artifacts are used in the paper.

in detail.

\subsection{Layer-by-Layer Analysis}

\subsubsection{L1 Contribution}
L1 provides the largest single reduction in ASR ($84\% \to 25\%$). As shown
in Tables~\ref{tab:breakdown-bare} and~\ref{tab:breakdown-l1}, L1 reduces
direct-injection ALLOW from 22 to 2, indirect/encoded from 22 to 2, and memory
poisoning from 16 to 1. It rejects most prompt injections before planning and
validates model outputs before any plan leaves the LLM-facing layer. However,
L1 remains a probabilistic filter: tool-poisoning ALLOW only drops from 24 to
20, because these requests use syntactically valid swap language to smuggle
malicious execution settings that a language-level filter cannot catch.

\subsubsection{L2 Contribution}
L2 provides the decisive final reduction ($25\% \to 0\%$). As shown in
Table~\ref{tab:breakdown-l1l2}, every adversarial case that survives L1 is now
caught by L2. It checks plans against deterministic rules such as token
allowlists, router allowlists, slippage limits, value caps, approval patterns,
quote freshness, and structural constraints. L2 is particularly important for
tool poisoning: under L1 alone, 20 of 25 tool-poisoning cases still ALLOW;
under L1+L2, all 20 are BLOCKED. A key architectural property is that L2 is
\textit{non-overridable} by LLM output. Even a complete jailbreak of the
planner cannot bypass the hard policy constraints once the plan reaches this
layer.

\subsubsection{L3 Contribution}
L3, the on-chain validator (\texttt{Swap\allowbreak Guard}), mirrors key L2
checks in an immutable contract and runs them through a pre-flight
\texttt{eth\_call}. Because L2 already achieves 0\% ASR on the benchmark, L3
does not further reduce the reported attack rate. Its value lies elsewhere: it
provides defense-in-depth against application-layer misconfiguration and gives
auditors a stable on-chain enforcement point independent of Python-side
configuration.

\subsection{Latency Considerations}

We report maximum request latency $\text{TR}$ for each configuration. The bare
configuration reaches 4.29\,s. L1 lowers the maximum observed latency to
3.14\,s, and L1+L2 reaches 3.08\,s. Enabling L3 does not create a meaningful
latency penalty; the L1+L2+L3 configuration records the lowest maximum latency
at 2.96\,s. These numbers are not statistical noise: early rejection via L1
\texttt{REFUSE} or L2 \texttt{BLOCK} short-circuits expensive LLM and tool
calls, so added defenses can improve both security and latency.

\subsection{Discussion}

The results support a core architectural principle for LLM-driven DeFi agents:
the LLM should be treated as an untrusted intent parser, not as a security
authority. The key boundary is a \textit{probabilistic-to-deterministic
handoff}: L1 performs stochastic language filtering, while L2 performs
deterministic policy enforcement on the critical path before the
owner-signature handoff.

The per-vector breakdown (Tables~\ref{tab:breakdown-bare}
through~\ref{tab:breakdown-l1l2}) provides the clearest evidence. Bare LLM
planning is unsafe: 84 of 100 adversarial cases produce ALLOW. Prompt-only
filtering (L1) is insufficient: tool poisoning remains almost entirely
unblocked (20/25 ALLOW). Deterministic policy enforcement (L2) closes the gap
completely. This layered architecture bounds the impact of any single failure:
even if an adversary compromises the LLM or poisons a tool response, the
deterministic policy engine still blocks the resulting plan before it can reach
the signer.

\section{Limitations}

Several limitations bound the scope of the present evaluation.

\textbf{Benchmark scope.} The evaluation covers only single-turn attacks.
Multi-turn conversational context poisoning, adaptive adversaries that refine
attacks based on prior rejections, and persistent state manipulation are not
represented in the current dataset.

\textbf{Canonical environment.} The main benchmark uses deterministic mock
market snapshots and quotes rather than live DEX APIs. This is a strength for
reproducibility, but it also means that real-world factors such as MEV
dynamics, network congestion, quote expiry variance, and aggregator-specific
edge cases are not part of the main reported numbers.

\textbf{Prototype signer bridge.} The current signer-boundary implementation is
an in-memory wallet handoff service rather than a production wallet SDK. It is
sufficient for demonstrating the pause-before-signature property, but not yet a
complete deployment-ready wallet integration.

\textbf{Prototype control-plane security.} The repository now supports optional
shared-secret protection on control-plane routes, but it does not yet implement
full production authentication, authorization, audit logging, or secret
rotation for those interfaces. This remains a research prototype rather than a
fully hardened service boundary.

\textbf{Static allowlists.} The token and router allowlists are small and fixed.
A production deployment would require continuous governance of these lists,
which introduces an additional operational attack surface.

\textbf{Narrow deployment scope.} The current asset universe and network scope
are intentionally narrow. Ethereum mainnet is the production target, Sepolia is
used for demo/test paths, and only a small allowlisted token/router set is
evaluated.

\textbf{Benign-set size.} The benign subset contains 25 cases. Achieving 0\%
false positives on this set is encouraging, but it still provides a limited
sample for estimating real-world usability at scale.

\textbf{L3 deployment assumptions.} L3 was evaluated through local Anvil-based
validation and demo-oriented public-testnet planning paths. Mainnet RPC
latency, contract upgrade governance, and operational maintenance costs are not
quantified here.

\textbf{Binary success metric.} ASR measures whether an unsafe plan is allowed,
but it does not capture the severity of financial loss that a successful attack
would inflict.

\section{Future Work}

Future work should address the limitations above in several directions:

\begin{enumerate}
    \item \textbf{Multi-turn adversarial dialogue.} Extend the benchmark to stateful, multi-turn interactions where adversaries gradually shift context or embed instructions across otherwise benign turns.
    \item \textbf{Systematic live adversarial evaluation.} Extend the existing real-tool smoke and guarded benchmark paths into a labeled adversarial benchmark against live DEX APIs, including quote-expiry races and tool-output poisoning attempts.
    \item \textbf{Production wallet and control-plane integration.} Replace the current in-memory handoff layer and shared-secret route protection with a production wallet bridge, persistent state, stronger authentication, and auditable operator workflows.
    \item \textbf{Formal and machine-checkable invariants.} Build on the current specification-driven traceability by modeling core L2 properties in a formal specification language such as TLA+.
    \item \textbf{Safe configuration governance.} Study how token and router allowlists should be updated safely, including review workflows, timelocks, parity checks, and monitoring of configuration drift between L2 and L3.
    \item \textbf{Broader deployment scope.} Evaluate the architecture on a wider token universe, richer benign workloads, and additional supported execution environments while preserving the same non-custodial boundary.
\end{enumerate}

\section{Conclusion}

This paper presented the design, implementation, and adversarial evaluation of a
minimal DeFi swap agent with layered security guardrails. The central argument
is that DeFi agents must treat LLM reasoning as untrusted and place
deterministic policy enforcement on the critical path before any owner
signature.

Empirical evaluation on 125 red-team cases supports this claim: an undefended
LLM planner allows 84\% of adversarial requests, while the L1+L2 configuration
reduces the attack success rate to 0\% without generating a single false
positive on benign traffic. The optional L3 on-chain validator adds
immutability and operational assurance, and its independent value is clearest
in the targeted misconfiguration experiment.

The non-custodial invariant remains the ultimate safety bound. Even if every
planning-layer control were to fail, the agent cannot sign or broadcast a
transaction. The owner must still review and authorize every plan, which bounds
the worst-case impact of any residual failure. As conversational interfaces
increasingly mediate financial workflows, this layered, non-custodial
architecture offers a practical path toward adversarially robust AI-driven
DeFi. In our prototype, that path is strengthened not only by benchmark
results, but also by engineering choices that make the boundary inspectable:
externalized prompt integrity, deterministic policy traces, explicit
owner-signature handoff state, live integration provenance, and parity checks
between the Python and Solidity layers.
