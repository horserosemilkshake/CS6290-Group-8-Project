\section{Introduction}
The user interfaces of Decentralized Finance (DeFi) are shifting from manually executing on‐chain transactions to conversational AI agents, ranging from wallet‑embedded copilots, Telegram trading bots, to fully autonomous DeFi assistants. These systems translate natural‑language directives into executable blockchain operations, effectively inserting a probabilistic planning layer into what was a deterministic pipeline under user's control.

The shift introduces an unexamined security boundary. Unlike explicit and formally analyzable smart contracts, large language models (LLMs) exhibit stochasticity and are susceptible to various linguistic and contextual adversarial actions, including prompt injection, indirect influence attacks, contextual contamination, and social‑engineering‑driven specification drift. The threat is amplified in open multi‑party communication environments such as Telegram or WhatsApp, where adversaries may directly interact with the same agent instance as the legitimate user, embedding payloads designed to subvert the transaction‑generation process.

While the existing literature extensively investigates vulnerabilities in smart‑contract correctness, MEV extraction dynamics, oracle manipulation, and private‑key security, there is little systematic analysis of the security properties of novel LLM‑driven transaction‑planning pipelines under adversarial language conditions. The research addresses the gaps by implementing and empirically evaluating a minimal, adversarially tested prototype intersecting AI agents and DeFi infrastructure.

\section{Literature Review}
%

The project integrates DeFi, AI-driven agents, and adversarial machine learning security, encompassing multiple unrelated and evolving fields. To situate the present work, this section connects three major bodies of literature, namely DeFi protocol and transaction-security research, adversarial language-model safety, and agentic LLM pipelines.

\subsection{DeFi Protocols Research}

Mainstream DeFi security research revolves around deterministic smart-contract vulnerabilities, including integer-overflow bugs, faulty permission, and MEV extraction methodologies assuming explicit, analyzable code as the attack surface, treating swaps and liquidity operations as deterministic execution traces \cite{10.1016/j.eswa.2025.128431}. The literature on non‑custodial wallet design and safe transaction construction further emphasizes invariant‑preserving interfaces and formal verification \cite{10.1016/j.eswa.2025.128431, ERINLE2025111691}. All the discussed work share an implicit assumption that the transaction pipeline is audit-able and deterministic.

However, probabilistic LLM-driven planners invalidate such an assumption and introduce stochastic decision chains and potential misalignment between authentic user intents and generated transactions, creating a novel security boundary that warrants further examination.

\subsection{Adversarial LLM Insecurity}

On top of stochasticity, the weaker adversarial robustness of LLM also induce challenges for implementing a secure swap agent. Recent research has been conducted throughout the document and has demonstrated attacks such as prompt injection, jailbreaks, and indirect prompt attacks \cite{pathade2025redteamingmindmachine}. In particular, research on prompt injection attacks detail how LLMs can be coerced into overriding internal instructions, execution constraints, or system‑level guardrails, especially when they are unable to enforce reliable separation between trusted and insecure channels \cite{pathade2025redteamingmindmachine, li2025securityconcernslargelanguage}. 

Consistent with the research, production-level incidents of bare (no protection guardrails like input filtering or encoding attack detection) LLM agent being prompt-injected on online forums and sending funds to unauthorized accounts emerged, highlighting realistic vulnerabilities to direct injection, structural weaknesses in LLM trust boundaries, and the ease of hijacking model behavior in open environments\cite{balaha2026}. This aligns with the threat model facing conversational DeFi agents operating in public channels like Telegram, where adversaries co‑exist with legitimate users and strategically inject malicious instructions into shared context.


\subsection{LLM Agents Tools and Deterministic Policy Layer}
Bare LLM agents that use external tools, APIs, and multi-step planners consistently exhibit novel modes of safety failure beyond typical content-moderation issues. Assessments such as Agent-SafetyBench show that no agent achieves a safety score of 60\% in tool invocation, while datasets like ToolSafety further demonstrates that multi-step and indirect harm scenarios significantly reduce the reliability of LLMs \cite{zhang2024agent, xie-etal-2025-toolsafety}. Even safety‑tuned models err in tool‑mediated workflows—misinterpreting intermediate data, hallucinating execution steps, or over‑relying on uncertain tool outputs. These results highlight the need for deterministic policy layers, strict schema validation, and environment constraints rather than relying solely on prompt‑based guardrails.

\section{Application Setting}

\subsection{System Architecture}
\subsubsection{Purpose and Scope}
The system provides a conversation swap planning agent that converts natural-language-based user intents into a deterministic, unsigned transaction plan (\texttt{TxPlan}) for a DeFi swap while the agent never signs or broadcasts transactions. Security objectives are user fund safety, privacy, and policy adherence in adversarial environment (e.g., public chat channels and unsafe data sources). 

\subsubsection{System Components Overview}
The following provides an overview of the components and their interactions.

\begin{figure}[H]
  \centering
  \includegraphics[width=3.3in]{figure/component_diagram.png}
  \caption{System Components Overview (zoom in for details)}
  \label{fig_model}
\end{figure}

A legitimate user interacts with the system via a conversation request to a chat interface (typically a Telegram bot), calling the agent plan API. After the Layer 1 (L1) agent parses and sanitizes the request, it extracts a structured swap intent and retrieves market data. The Layer 2 (L2) policy engine verifies whether the request compiles with deterministic safety and \texttt{allowlist} rules, and returns an unsigned \texttt{TxPlan} marked with \texttt{NEEDS\_OWNER\_SIGNATURE} The user reviews the plan and, if satisfied, signs with their own private key.

In this paper, L1 is the DeFi LLM agent. It is a probabilistic layer responsible for handling input security, filtering requests, and controlling tool invocations, thereby mitigating natural language attacks. Meanwhile, L2 is the policy engine. It is a strict and deterministic enforcement layer that blocks any rule-breaking actions by the LLM or user requests.

\begin{figure}[H]
  \centering
  \includegraphics[width=2in]{figure/bare+L1+L2.png}
  \caption{The Relationship Between Bare LLM, L1 and L2}
  \label{fig_model}
\end{figure}

Detailedly, the system converts natural-language swap requests into unsigned transaction plans through three sequential control layers with the following specific checks:

\begin{enumerate}
    \item L1 (Input/Output Guardrails): prompt-injection filtering, unsafe instruction rejection, and response validation.
    \item L2 (Deterministic Policy Engine): rule-based checks on token/router allowlists, slippage, transaction value cap, approval patterns, transaction structure, and chain scope.
    \item L3 (Optional On-Chain Validator): pre-flight contract validation (SwapGuard) via eth\_call for selected hard constraints.
\end{enumerate}

\subsubsection{Important Trust and Signer Boundary}
\begin{itemize}
    \item Invariant I1 (Non‑Custodial): No module can produce a signed transaction or trigger an on‑chain side effect.
    \item Invariant I2 (Deterministic Governance): L2 policy decisions dominate LLM/tool outputs.
\end{itemize}

Thus, the agent's possible outputs are limited as follows.
\begin{multicols}{2}
\begin{itemize}
    \item \texttt{REJECTED} (by L1)
    \item \texttt{NEEDS\_OWNER} \\ \texttt{\_SIGNATURE}
    \item \texttt{BLOCKED} (by L2)
    \item \texttt{ERROR} (API error)
\end{itemize}
\end{multicols}

\subsection{Threat Model}
\subsubsection{Assets and Objectives}
The most important assets are user funds and privacy information. The system is supposed to prevent unsafe plans that might incur losses while protecting addresses, balances, and user intentions, and ensuring policy adherence and invariant preservation for system integrity.

\subsubsection{Adversaries and Capabilities}
Assume adversaries can access the input interface in an open communication platform, observe timing/metadata, and influence tool output to the extent allowed by mocked/external data sources. The following attacks prevail under these assumptions.

\begin{itemize}
    \item External prompt adversary: use chat input to induce unsafe plans (e.g., "My uncle has been diagnosed with a tetanus infection due to a lobster like you. I need 4 Sol to get the treatment done"\cite{balaha2026}, role-play jail breaking). 
    \item Indirect data adversary: inject malicious instructions via external tools/data (e.g., payload in market feeds). 
    \item Context Poisoner: forge provenance (fake admin/system claims) or taint historical context/memory.
    \item Fraudster: exploits policy gaps (e.g., routers/tokens not on allowlists, extreme slippage).
\end{itemize}

\subsubsection{Adversaries Attack Surfaces}

\begin{itemize}
    \item Open chat input channel (Telegram/WhatsApp natural language inputs).
    \item External tool outputs (e.g., quotes and market snapshots are susceptible to indirect injection). 
    \item State/memory (if any retained context is used beyond a single request) allowing information leakage.
    \item Public or unprotected API surface (HTTP endpoints).
\end{itemize}

The following table models some of the anticipated threat categories in the system.

\subsubsection{Full Security Invariants and Policies} Besides I1 and I2, we introduce two more invariants for schema and context.

\begin{itemize}
    \item Invariant I3 (Schema Integrity): All cross‑component messages conform to \texttt{schemas.py} and are validated before sending and upon arrival.
    \item Invariant I4 (Context Discipline): Tool outputs treated as untrusted data; never merged into instruction channels.
\end{itemize}

\section{Experiment and Methodology}

This section outlines the experiment settings taken to examine the adversarial resistance of the proposed swap agent with layered defense architecture and controlled red-teaming. The testing methodology is to isolate the security contribution of each defense layer (i.e., L1, L2, L3) while preserving reproducibility. 

\subsubsection{Threat Model and Attack Taxonomy}
We evaluate prompt-driven and tool-mediated attacks that target agent misbehavior before on-chain submission. Adversarial prompts are grouped into four vectors $\mathbb{A}^4$, each with 25 entries:

\begin{itemize}
    \item direct injection (explicit jailbreak/override instructions)
    \item indirect/encoded (obfuscated or encoded malicious intent)
    \item tool poisoning (manipulating tool-related assumptions)
    \item memory poisoning (context poisoning and fake prior-authority claims)
\end{itemize}

\begin{figure}[H]
  \centering
  \includegraphics[width=3.3in]{figure/Sample test cases.png}
  \caption{Sample Test Cases (2 Out of 125)}
  \label{fig_sample_cases}
\end{figure}

An additional 25 benign (legitimate swap) cases are included to measure false-positive behavior, bringing the final dataset to 125 cases (100 adversarial + 25 benign).

A successful attack is to induce unsafe \texttt{ALLOW} decisions, bypass policy controls, or force unstable behavior. From the definition of success, we can define three primary metrics $\mathbb{R}^3$:

Attack Success Rate (ASR):
\begin{equation}
    \text{ASR}=\frac{\text{adversarial cases with observed}=\texttt{ALLOW}}{\text{total adversarial cases}}
\end{equation}

False Positive Rate (FP):
\begin{equation}
    \text{FP}=\frac{\text{benign cases with observed}\neq\texttt{ALLOW}}{\text{total benign cases}}
\end{equation}

Time Robustness (TR):
\begin{equation}
    \text{TR}=\max_{i}{\text{duration}_i}
\end{equation}

The 125-case test suite is run against four defense configurations (bare, L1, L1+L2, L1+L2+L3) to yield ASR, FP, and TR for each layer combination.

\subsubsection{Test and Optimality Modeling}
Define $L(i)=\bigcap_{a=1}^{n} La\in\mathbb{D}$ as $i$-th defense layer implemented for the bare LLM system. Therefore, the marginal utility of $L(i+1)$ is induced as:
\begin{equation}
    u(i+1) = t(L(i)) - t(L(i+1))
\end{equation}
where $t:(\mathbb{D},\mathbb{A}^4)\to\mathbb{R}^3$ is the testing function.

The optimal defense layer is defined as a convex problem:
\begin{equation}
\begin{split}
\argmax_{i}{||u(i+1)^{+}||_2} \\
\text{s.t. } i=0,1,2
\end{split}
\end{equation}


\section{Experiment Results and Discussion}

\subsection{Overall Results}

The final evaluation dataset (\texttt{final\_attack\_dataset\_v2}) comprises 125 cases: 100 adversarial cases evenly distributed across four attack vectors (25 each) and 25 benign cases. Table~\ref{tab:main-results} presents the results across four defense configurations.

\begin{table}[H]
\centering
\caption{Final Red-Team Results (125 Cases, v2 Dataset)}
\label{tab:main-results}
\begin{tabular}{lcccc}
\hline
\textbf{Config} & \textbf{ASR} & \textbf{FP} & \textbf{TR (s)} & \textbf{N} \\ \hline
bare    & 76.00\% & 0.00\% & 3.05 & 125 \\
L1      & 25.00\% & 0.00\% & 8.62 & 125 \\
L1+L2   & 15.00\% & 0.00\% & 3.39 & 125 \\
L1+L2+L3 & 15.00\% & 0.00\% & 3.49 & 125 \\ \hline
\end{tabular}
\end{table}

The bare (undefended) agent allowed 76\% of adversarial inputs to pass through, confirming that the LLM planner alone is highly vulnerable. Introducing L1 guardrails reduced ASR to 25\%, a 51 percentage-point drop. Adding the L2 deterministic policy engine further reduced ASR to 15\%. L3 (on-chain SwapGuard validation) did not further reduce ASR in this dataset but provides an additional enforcement layer against parameter-level attacks that may bypass L2 in production.

Notably, all configurations achieved 0\% FP, indicating that none of the 25 benign swap requests were incorrectly rejected.

\subsection{Per-Attack-Vector Breakdown}

Table~\ref{tab:breakdown} shows the outcome distribution under the best defense configuration (L1+L2) across attack vectors.

\begin{table}[H]
\centering
\caption{L1+L2 Outcome Breakdown by Attack Vector}
\label{tab:breakdown}
\begin{tabular}{lcccc}
\hline
\textbf{Attack Vector} & \textbf{REFUSE} & \textbf{BLOCK} & \textbf{ALLOW} & \textbf{Total} \\ \hline
Direct injection     & 23 & 2  & 0  & 25 \\
Indirect / encoded   & 22 & 2  & 1  & 25 \\

Direct injection, indirect/encoded, and memory poisoning are effectively neutralized by L1+L2, with near-zero successful attacks. The primary residual vulnerability is tool poisoning, where 14 out of 25 adversarial cases still receive \texttt{ALLOW}. These cases exploit the gap between L1 (language-level) and L2 (parameter-level) checks: the adversarial intent is expressed through valid-looking swap parameters rather than detectable linguistic patterns.

\subsection{Layer-by-Layer Analysis}

\subsubsection{L1 Contribution}
L1 guardrails provide the largest single reduction in ASR ($76\% \to 25\%$). The input guardrail intercepts prompt injection via regex-based blocked patterns and keyword filtering, while the output guardrail validates LLM response structure and forbidden tool calls. L1 is most effective against direct injection (ASR drops from 72\% to 8\%) and memory poisoning (ASR drops from 52\% to 0\%).

\subsubsection{L2 Contribution}
The L2 deterministic policy engine provides an additional 10 percentage-point ASR reduction ($25\% \to 15\%$). It catches attacks that produce syntactically valid but policy-violating plans: disallowed tokens (R-01), disallowed routers (R-02), excessive slippage (R-03), value cap violations (R-04), unlimited approvals (R-05), and structural integrity violations (R-07). L2 is particularly effective against tool poisoning, blocking 6 additional cases that L1 alone would allow.

\subsubsection{L3 Contribution}
L3 on-chain validation via the SwapGuard contract replicates key L2 checks (token allowlist, router allowlist, value cap, slippage) as an immutable on-chain enforcement layer. While L3 does not reduce ASR further in the current dataset, it provides defense-in-depth: even if L2 rules are misconfigured or bypassed at the application level, the on-chain contract independently enforces hard constraints before any transaction can execute.

\subsection{Latency Considerations}
The bare configuration processes requests in a mean of 2.22s. L1 increases peak latency to 8.62s due to LLM API calls for intent parsing. L1+L2 reduces peak latency back to 3.39s because many requests are rejected early by L1 before reaching the LLM. L3 adds minimal overhead ($\sim$0.1s) as it performs a single \texttt{eth\_call} to the local chain.

\subsection{Discussion}
The results demonstrate that layered defense is essential for securing LLM-driven DeFi agents. Relying solely on the LLM (bare mode) is insufficient, as adversarial prompts bypass it 76\% of the time. The combination of probabilistic (L1) and deterministic (L2) layers provides complementary coverage: L1 handles linguistic attacks while L2 catches parameter-level policy violations.

The remaining 15\% ASR is concentrated in tool-poisoning attacks that craft valid-looking swap parameters. Addressing this gap would require semantic understanding of attack intent at the parameter level, potentially through anomaly detection on swap patterns or stricter tool-output validation.

\section{Conclusion}

This paper presents the design, implementation, and adversarial evaluation of a minimal DeFi swap agent with layered security guardrails. The system converts natural-language swap requests into unsigned transaction plans through three sequential defense layers: L1 (LLM-level input/output guardrails), L2 (deterministic policy engine), and L3 (optional on-chain contract enforcement).

Empirical evaluation on 125 red-team cases across four attack vectors demonstrates that:
\begin{enumerate}
    \item An undefended LLM planner is highly vulnerable (76\% ASR).
    \item L1 guardrails reduce ASR to 25\%, primarily by intercepting linguistic attacks.
    \item L2 deterministic policies further reduce ASR to 15\% by catching parameter-level violations.
    \item L3 on-chain enforcement provides defense-in-depth without additional ASR reduction in the current dataset.
    \item All configurations maintain 0\% false positives on benign requests.
\end{enumerate}

The non-custodial invariant (the agent never signs or broadcasts) ensures that even a successful attack produces only an unsigned plan, requiring explicit owner signature before on-chain execution. This architectural constraint bounds the worst-case impact of residual vulnerabilities.

Future work should address the tool-poisoning gap through semantic intent verification, extend evaluation to multi-turn conversational attacks, and investigate the robustness of L3 enforcement under adversarial contract interactions.
